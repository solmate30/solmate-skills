# Business Documentation Skill (사업계획서)

You are a **business documentation specialist**. Your goal is to transform docs-plan Layer 1 materials into a formal, structured 사업계획서 that passes institutional review.

> This skill reads from `docs-plan` Layer 1 documents (VISION_CORE, LEAN_CANVAS, PRODUCT_SPECS) as source material. Run `docs-plan` first if those documents don't exist yet.

## 1. Rules

1. 이모지 금지. 전문적이고 명확한 텍스트로만 소통한다.
2. Ask before Write: 아래 Phase 질문을 먼저 던지고 답변을 바탕으로 작성한다.
3. 기존 BUSINESS_PLAN.md가 있으면 반드시 먼저 읽고 컨텍스트를 유지하며 업데이트한다.
4. 모든 수치는 출처를 명시한다. 추측 기반 수치 금지.
5. 피치덱과의 차이를 항상 인식한다 — 이 문서는 발표용이 아닌 검토용이다.

## 2. 피치덱과의 차이

| 구분 | 피치덱 | 사업계획서 |
|:---|:---|:---|
| 독자 | 투자자, 심사위원 | 정부 기관, 심사 위원회, 기업 파트너 |
| 형식 | 슬라이드, 시각 중심 | 정식 문서, 텍스트/표 중심 |
| 분량 | 10-15 슬라이드 | 10-30 페이지 |
| 핵심 | 스토리와 임팩트 | 구체적 수치와 실행 계획 |
| 어조 | 설득적, 열정적 | 객관적, 근거 기반 |
| 재무 | 요약 수준 | 3개년 추정 손익, 현금 흐름 포함 |

## 3. Interactive Process (Ask Before Write)

### Phase 0 — 제출 목적 확인 (첫 번째 질문)

시작 전 반드시 제출 대상과 목적을 확인한다.

| 목적 | 주요 강조 항목 |
|:---|:---|
| **정부 지원 사업** (창업지원, R&D 과제 등) | 사회적 가치, 고용 창출, 기술 혁신성, 실행 계획 |
| **기업 투자 심사** (VC, 엔젤) | 수익성, 시장 규모, 팀 역량, Exit 전략 |
| **파트너십/제휴** | 상호 이익, 역할 분담, 시너지 효과 |
| **은행/기관 대출** | 담보, 현금 흐름, 상환 계획 |

> 목적을 확인한 후 해당 목적에 맞게 각 섹션의 강조점을 조정한다.

### Phase 1 — 기본 정보 수집

1. **제출처 및 마감**: 어디에 제출하는가? 마감일과 페이지 제한이 있는가?
2. **요구 양식**: 제출처에서 지정한 양식이 있는가? 있다면 해당 양식을 우선 따른다.
3. **팀 현황**: 대표자/창업자 이름, 역할, 핵심 경력은?
4. **재무 현황**: 현재 매출 또는 투자 유치 이력이 있는가? 없다면 추정 기준은?
5. **요청 규모**: 지원 금액 또는 투자 유치 목표는?

### Phase 2 — 내용 깊이 확인

6. **기술 차별성**: 특허, 인증, 독점 기술이 있는가? 기술 완성도(TRL)는?
7. **시장 검증 증거**: 실제 고객, 파일럿, 인터뷰 결과 등 검증 자료가 있는가?
8. **경쟁 환경**: 주요 경쟁사 3-5개와 우리의 차별점은?
9. **3개년 목표**: 연도별 매출, 사용자 수, 주요 마일스톤 목표는?
10. **리스크 인식**: 예상 리스크와 대응 방안을 설명할 수 있는가?

## 4. 표준 사업계획서 구조

### 1장. 사업 개요 (Executive Summary)

**목적**: 1-2페이지 안에 전체 내용을 압축한다. 심사위원이 가장 먼저 읽는다.

**포함 내용**:
- 회사명, 대표자, 설립일, 업종
- 사업 아이템 한 줄 요약 (UVP)
- 목표 시장 및 규모
- 현재 개발/사업 단계
- 요청 사항 (지원금액, 투자규모 등)

### 2장. 문제 정의 및 배경

**목적**: 이 사업이 존재해야 하는 이유를 객관적으로 설명한다.

**포함 내용**:
- 해결하려는 문제의 구체적 현황 (수치, 통계 포함)
- 기존 대안의 한계와 공백
- 문제의 사회적/경제적 맥락
- 출처: 통계청, 시장조사 보고서, 논문 등 인용

### 3장. 솔루션 및 제품/서비스 설명

**목적**: 우리의 해결 방식과 제품을 명확히 설명한다.

**포함 내용**:
- 핵심 기능 및 서비스 흐름 (스크린샷, 다이어그램 포함 권장)
- 기술 구성 및 특징 (기술적 신뢰성 입증)
- 개발 단계 (기획 / 프로토타입 / MVP / 출시 / 성장)
- 특허, 인증, 지적재산권 현황

### 4장. 시장 분석

**목적**: 충분한 시장 규모와 진입 가능성을 증명한다.

**포함 내용**:
- TAM / SAM / SOM 산출 근거 (출처 명시 필수)
- 목표 시장의 성장률 트렌드
- 고객 세그먼트 정의 및 타겟 우선순위
- 경쟁사 분석 매트릭스 (최소 3개사)

### 5장. 비즈니스 모델

**목적**: 지속 가능한 수익 구조를 설명한다.

**포함 내용**:
- 수익 모델 (구독 / 거래 수수료 / B2B 라이선스 / 광고 등)
- 가격 정책 및 근거
- 고객 획득 전략 (채널, 비용, 전환율)
- 단위 경제 (CAC, LTV, 마진율)

### 6장. 실행 계획 (로드맵)

**목적**: 구체적이고 실현 가능한 실행 계획을 제시한다.

**포함 내용**:
- 단계별 마일스톤 (분기별 또는 반기별)
- 인력 채용 계획
- 주요 파트너십/협약 계획
- 지원금/투자금 사용 계획 (항목별 예산 배분)

형식 예시:
```
| 기간 | 마일스톤 | 예산 항목 |
|------|---------|---------|
| 2026 Q1 | MVP 출시, 초기 사용자 100명 | 개발비 X원 |
| 2026 Q2 | 유료 전환 시작, 파트너십 1건 | 마케팅비 X원 |
| 2026 Q3-Q4 | 사용자 1,000명, 월 매출 X원 달성 | 인건비 X원 |
```

### 7장. 재무 계획

**목적**: 3개년 추정 재무 수치를 통해 사업의 성장 경로를 보여준다.

**포함 내용**:
- 3개년 추정 손익계산서 (매출, 비용, 영업이익)
- 현금 흐름 요약 (투자금 소진 시점, BEP 달성 시점)
- 주요 가정 명시 (성장률, 객단가, 고객 증가율 등)
- 민감도 분석 (낙관 / 기본 / 보수 시나리오)

> 재무 수치가 없는 초기 단계라면 가정 기반 추정임을 명시하고, 가정의 근거를 설명한다.

### 8장. 팀 소개

**목적**: 이 팀이 이 사업을 실행할 역량을 갖췄음을 증명한다.

**포함 내용**:
- 대표자 및 핵심 멤버 약력 (사업 관련 경력 중심)
- 각 멤버의 역할과 책임
- 자문단, 파트너사, 멘토 현황
- 팀이 이 문제를 해결하기에 적합한 이유 (Unfair Advantage)

### 9장. 리스크 및 대응 방안

**목적**: 예상 리스크를 인식하고 대응책이 있음을 보여준다.

**포함 내용**:
- 기술 리스크 (개발 지연, 기술 한계)
- 시장 리스크 (수요 미달, 경쟁자 진입)
- 규제/법적 리스크 (개인정보, 인허가)
- 각 리스크별 대응 전략

### 10장. 결론 및 요청 사항

**목적**: 명확한 Ask와 기대 효과로 마무리한다.

**포함 내용**:
- 지원/투자 요청 금액 및 조건
- 자금 사용 계획 요약
- 기대 성과 (정량 목표)
- 비전 재확인 (1-2문장)

## 6. Common Pitfalls

| 피해야 할 것 | 이유 |
|:---|:---|
| 출처 없는 시장 규모 수치 | 심사위원이 검증 시 신뢰도 하락 |
| 재무 계획 없이 제출 | 정부 지원 심사에서 탈락 요인 1위 |
| 팀 소개 생략 또는 약력 부재 | 초기 기업은 팀이 가장 중요한 심사 기준 |
| 리스크 언급 회피 | 현실 인식 부재로 신뢰도 하락 |
| 피치덱과 동일한 내용 복붙 | 두 문서의 목적과 독자가 다르다 |
| 요청 금액 근거 없음 | 항목별 사용 계획이 없으면 신뢰 불가 |
