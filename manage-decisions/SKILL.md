# 결정 관리 스킬 (manage-decisions)

프로젝트를 진행하다 보면 AI가 독단적으로 결정해서는 안 되는 순간들이 있습니다. 이 스킬은 그 순간을 인식하고, 결정 유형에 맞는 질문을 사용자에게 던져 충분한 대화와 맥락을 바탕으로 결정이 이루어지도록 합니다.

**원칙**: 결정을 내리기 전에 먼저 묻는다. 추측하거나 "일반적으로는" 식의 답변으로 진행하지 않는다.

---

## 실행 시점

- "이거 어떻게 할까요?", "어떤 방식이 좋을까요?" 질문을 받았을 때
- 기술 스택·라이브러리·패턴을 선택해야 할 때
- 기능 범위를 정의하거나 MVP 경계를 정해야 할 때
- DB 스키마·API 구조·아키텍처 설계를 시작하기 전
- `rules-workflow` Step 1~4 (계획 수립·검토) 수행 중 결정이 필요할 때
- `rules-product` 각 Phase 진입 전 핵심 방향을 잡을 때

---

## Step 0: 결정 유형 파악

사용자의 요청 또는 현재 맥락에서 아래 6개 유형 중 하나를 먼저 식별한다. 유형이 복수인 경우 우선순위가 높은 것부터 순서대로 진행한다.

| 유형 | 키워드 | 대표 상황 |
|------|--------|-----------|
| A. 기술 선택 | 스택, 라이브러리, 프레임워크, 도구 | "ORM은 뭘 쓸까요?", "상태관리는?" |
| B. 기능 범위 | MVP, 기능, 우선순위, 범위, 스코프 | "이 기능도 넣을까요?", "MVP 범위는?" |
| C. DB 설계 | 테이블, 스키마, 관계, 필드, 인덱스 | "테이블을 어떻게 나눌까요?" |
| D. API 설계 | 엔드포인트, REST, 응답 구조, 인증 | "API 구조를 어떻게 잡을까요?" |
| E. UX·UI 방향 | 화면, 흐름, 레이아웃, 인터랙션 | "이 화면은 어떻게 구성할까요?" |
| F. 아키텍처 | 구조, 레이어, 모듈, 배포, 확장성 | "폴더 구조를 어떻게 잡을까요?" |

---

## Type A: 기술 선택

**목적**: 필연적인 기술 선택. "일반적으로 많이 쓰여서"가 아니라 프로젝트 요구사항에서 이 기술이 필요한 이유를 먼저 확인한다.

**질문 순서**:

1. **제약 조건 먼저**: 현재 프로젝트의 기술 스택(언어, 런타임, 배포 환경)에서 이미 확정된 것은 무엇인가?
2. **핵심 요구사항**: 이 기술을 선택하는 핵심 기능적 이유는 무엇인가? (예: 실시간 필요 → WebSocket, 타입 안전 DB 쿼리 필요 → Drizzle/Prisma)
3. **트레이드오프 인식**: 후보 옵션들의 장단점을 비교해도 되는가? (예: A는 기능이 많지만 번들이 크고, B는 가볍지만 커뮤니티가 작다)
4. **팀 친숙도**: 현재 팀(또는 본인)이 이 기술에 얼마나 익숙한가? 학습 비용이 허용되는가?
5. **미래 의존성**: 이 선택이 나중에 교체하기 어려운 결정인가? (예: DB 종류, 인증 라이브러리)

**결정 기록 형식**:
```
결정: [선택한 기술]
이유: [핵심 이유 1-2줄]
포기한 대안: [대안] — [포기 이유]
재검토 조건: [이 결정을 바꿔야 하는 상황]
```

---

## Type B: 기능 범위

**목적**: AI가 임의로 기능을 추가하거나 제외하지 않도록 범위를 사용자와 함께 확정한다.

**질문 순서**:

1. **현재 단계 확인**: 지금이 MVP인가, 이후 버전인가? Phase 1(기획) 문서에 정의된 MVP 범위가 있는가?
2. **Must Have vs Nice to Have**: 이 기능이 없으면 핵심 가치(UVP)가 전달되지 않는가, 아니면 있으면 좋은 수준인가?
3. **사용자 여정 필수성**: 이 기능 없이 사용자가 핵심 목표를 달성할 수 없는가?
4. **구현 비용 인식**: 이 기능의 구현 복잡도가 다른 기능들에 비해 어느 수준인가? 이 비용이 가치에 비례하는가?
5. **제외 시 영향**: 이 기능을 이번 버전에서 제외하면 어떤 문제가 생기는가?

**결정 기록 형식**:
```
기능: [기능명]
결정: 포함 / 제외 / 다음 버전
이유: [결정 근거]
제외 시 대안: [없으면 "없음"]
```

---

## Type C: DB 설계

**목적**: 스키마 결정은 나중에 바꾸기 어렵다. 충분한 논의 없이 진행하지 않는다.

**질문 순서**:

1. **데이터 성격**: 이 데이터는 관계형(정형)인가, 비정형인가? 어떤 쿼리 패턴이 가장 빈번한가?
2. **소유권과 관계**: 이 엔티티는 누가 소유하는가? (User → Post → Comment 같은 계층 구조가 있는가?)
3. **확장 가능성**: 이 테이블에 나중에 어떤 필드가 추가될 가능성이 높은가? 지금 설계가 그것을 수용하는가?
4. **삭제 정책**: 레코드 삭제 시 `hard delete`인가, `soft delete`(deleted_at)인가? 관련 데이터는 어떻게 처리하는가?
5. **인덱스 우선순위**: 가장 자주 조회되는 필드 조합은 무엇인가? (검색, 필터, 정렬 기준)
6. **민감 데이터**: 이 테이블에 PII(개인식별정보)가 포함되는가? 암호화·접근 제한이 필요한가?

**결정 기록 형식**:
```
테이블: [테이블명]
주요 결정: [핵심 설계 선택사항]
이유: [결정 근거]
주의사항: [나중에 재검토가 필요한 부분]
```

---

## Type D: API 설계

**목적**: 클라이언트-서버 계약(contract)은 한번 노출되면 바꾸기 어렵다. 구현 전에 구조를 먼저 합의한다.

**질문 순서**:

1. **클라이언트 요구사항**: 이 API를 호출하는 클라이언트는 무엇인가? (웹/앱/외부 파트너) 각자 필요한 데이터 형태가 다른가?
2. **인증 방식**: 이 엔드포인트는 인증이 필요한가? 어떤 인증 방식을 사용하는가? (세션, JWT, API Key)
3. **응답 구조**: 성공·실패 응답의 표준 형식이 있는가? 페이지네이션이 필요한가?
4. **버전 관리**: API 버전 관리 전략이 있는가? (`/api/v1/`) 현재 명세 변경이 기존 클라이언트에 영향을 주는가?
5. **에러 처리**: 어떤 에러 케이스가 예상되는가? 클라이언트가 각 에러를 어떻게 처리해야 하는가?
6. **부작용 범위**: 이 API 호출이 다른 데이터에 side effect를 일으키는가? (예: 주문 생성 → 재고 차감 → 알림 발송)

**결정 기록 형식**:
```
엔드포인트: [METHOD /path]
결정: [설계 선택사항]
이유: [결정 근거]
미확정 사항: [추후 확인 필요한 부분]
```

## Type E: UX·UI 방향

**목적**: 화면 구성·흐름·인터랙션은 사용자 경험에 직결된다. AI가 임의로 레이아웃을 결정하지 않는다.

**질문 순서**:

1. **사용자 목표**: 이 화면에서 사용자가 달성하려는 핵심 목표는 무엇인가? 가장 빈번한 액션은?
2. **진입 경로**: 사용자가 이 화면에 어떻게 도달하는가? 이전 화면의 맥락이 이 화면에 영향을 주는가?
3. **정보 우선순위**: 이 화면에서 가장 먼저 눈에 들어와야 하는 요소는 무엇인가? (CTA, 핵심 정보, 상태)
4. **빈 상태(Empty State)**: 데이터가 없을 때 어떻게 보여야 하는가? 사용자에게 다음 행동을 유도하는가?
5. **에러·로딩 상태**: 로딩 중, 에러 발생 시 화면을 어떻게 처리하는가?
6. **모바일 우선 여부**: 이 화면은 모바일에서 더 많이 사용되는가? 반응형 레이아웃 전략이 있는가?

**결정 기록 형식**:
```
화면: [화면명]
핵심 결정: [레이아웃·흐름·인터랙션 선택]
이유: [결정 근거]
확인 필요: [사용자 테스트 또는 추가 검토 필요한 부분]
```

---

## Type F: 아키텍처

**목적**: 폴더 구조, 레이어 분리, 모듈 경계는 팀 전체 생산성에 영향을 준다. 한 명이 독단적으로 결정하지 않는다.

**질문 순서**:

1. **팀 규모와 협업 방식**: 이 프로젝트는 혼자 진행하는가, 팀으로 진행하는가? 도메인별로 역할이 나뉘는가?
2. **성장 예측**: 6개월 후 코드베이스 규모가 지금의 몇 배가 될 것 같은가? 현재 구조가 그것을 수용하는가?
3. **레이어 경계**: 비즈니스 로직, 데이터 접근, UI를 어디서 분리할 것인가? (예: `services/`, `repositories/`, `components/`)
4. **공유 코드 전략**: 여러 페이지·기능에서 공통으로 쓰이는 코드(유틸, 훅, 타입)를 어디에 두는가?
5. **의존성 방향**: 모듈 간 의존성이 단방향인가? 순환 의존성이 발생할 가능성이 있는가?
6. **배포 단위**: 모노레포인가 멀티레포인가? 프론트·백을 분리 배포하는가?

**결정 기록 형식**:
```
결정: [아키텍처 선택사항]
이유: [결정 근거]
전제 조건: [이 구조가 성립하는 조건]
재검토 시점: [이 결정을 다시 봐야 하는 상황]
```

## 결정 기록 저장

중요한 결정은 다음 경로에 누적 기록한다:

```
docs/04_Logic_Progress/03_DECISION_LOG.md
```

형식:
```markdown
# Decision Log
> Created: YYYY-MM-DD HH:mm
> Last Updated: YYYY-MM-DD HH:mm

## [YYYY-MM-DD] [결정 제목]
- **유형**: A·B·C·D·E·F 중 하나
- **결정**: [선택한 방향]
- **이유**: [핵심 근거]
- **포기한 대안**: [대안 및 이유]
- **재검토 조건**: [이 결정을 바꿔야 하는 상황]
```

## Anti-Rush (AGENTS.md 준수)

- 사용자가 "그냥 알아서 해줘"라고 해도 중요한 결정은 질문을 먼저 한다.
- 답변이 불충분하면 보완 질문을 한다. 추측으로 채우지 않는다.
- 결정이 내려지면 반드시 요약하여 사용자에게 확인받은 후 다음 단계로 넘어간다.
- 결정 기록을 `03_DECISION_LOG.md`에 남기도록 안내한다.
